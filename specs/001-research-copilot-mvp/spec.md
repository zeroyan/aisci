## 阶段目标

本阶段只交付一个最小可运行闭环：

1. 输入 `ResearchSpec/ExperimentPlan`。
2. AI 自主执行”生成/修改代码 -> 执行 -> 分析 -> 再决策”。
3. 命中停止条件后输出 `ExperimentReport`。
4. 全程产物可追溯、可复现。

## 输入来源

Phase 1A 不负责生成 `ResearchSpec` 或 `ExperimentPlan`（这是 Phase 1B/1C 的工作）。
本阶段的输入通过以下方式提供：

1. **手动种子数据**：用户手写 JSON 文件放入 `runs/<run_id>/spec/research_spec.json`
   和可选的 `runs/<run_id>/plan/experiment_plan.json`，格式满足 contracts 即可。
2. **CLI 参数**：通过命令行传入 spec 文件路径。
3. **测试用例**：仓库预置 `tests/fixtures/` 下的样例 spec 和 plan，用于开发和 CI。

只要输入满足 contracts 中 `ResearchSpec(status=confirmed)` 的约束，系统即可启动 run。

## In Scope（Phase 1A）

1. AI 在沙箱中多轮实验迭代。
2. 预算、时限、人工停止的硬中止能力。
3. 每轮实验日志、参数、结果和成本记录。
4. 运行结束后输出结构化实验报告与证据映射。

## Out of Scope（留给后续阶段）

1. 多源选题与候选课题评估（Phase 1C）。
2. 方案设计与 baseline 自动验证（Phase 1B）。
3. 论文自动写作、自动审稿、复杂记忆沉淀（P2）。

## 用户故事与验收

### 用户故事 1（US-1）— AI 自主实验执行

作为科研用户，我希望 AI 能自主写代码、跑实验、分析结果并持续改进，直到达到停止条件或需要我介入。

**验收场景**：
1. **Given** 已确认的 `ResearchSpec` 或 `ExperimentPlan`，
   **When** 启动 run，
   **Then** 系统在沙箱中完成首轮实验并输出结构化指标。
2. **Given** 结果未达标，
   **When** 系统完成本轮分析，
   **Then** 系统自动产生下一轮改进动作并继续执行。
3. **Given** 命中预算/时限/人工停止，
   **When** run 终止，
   **Then** 所有中间结果完整落盘并可复查。
4. **Given** 多轮运行结束，
   **When** 生成总结报告，
   **Then** 每条结论都能映射到具体证据文件。

## 边界情况

1. LLM 超时或限流：重试后仍失败则暂停并提示人工介入。
2. 代码运行崩溃：保留崩溃日志，允许自动修复重试。
3. 连续失败超过阈值：暂停并请求人类决策。
4. 预算耗尽或时限到达：立即硬停并落盘当前状态。

## 功能需求（Phase 1A）

> 本地编号 `1A-FR-xxx`，括号内为全局 Epic 编号。

- **1A-FR-001**（E-FR-008, E-FR-009）: 系统必须在沙箱中执行 AI 自主实验循环（生成/修改代码、运行、分析、再决策）。
- **1A-FR-002**（E-FR-010）: 系统必须支持多轮迭代，并记录每轮代码变更、参数、指标、日志。
- **1A-FR-003**（E-FR-013）: 系统必须支持预算、时限、人工停止三类硬停止条件。
- **1A-FR-004**（E-FR-012）: 系统必须在不可恢复失败时暂停并请求人工介入。
- **1A-FR-005**（E-FR-020）: 系统必须为每次运行创建 `runs/<run_id>/` 并保存全部工件。
- **1A-FR-006**（E-FR-010 补充）: 系统必须输出 `ExperimentReport`，并提供”结论 -> 证据路径”映射。
- **1A-FR-007**（E-FR-022）: 系统必须记录模型调用次数、token 消耗、估算成本。
- **1A-FR-008**（E-FR-023）: 系统必须符合 `contracts.md` 定义的数据、状态机、错误契约。
- **1A-FR-009**（E-FR-011）: 系统必须在实验结果未达预期时自动分析原因并调整方案，无需人类干预。

## 关键实体

- **ExperimentIteration**: 单轮实验，包含代码变更、执行命令、指标、日志、状态。
- **ExperimentRun**: 完整运行，包含多轮迭代、终止原因、成本统计。
- **ExperimentReport**: 最终报告，包含最佳结果、失败尝试、证据映射。

## 成功指标（Phase 1A）

- **SC-001**: 单次 run 可连续完成至少 3 轮自主迭代（资源允许时）。
- **SC-002**: 预算/时限触发时，系统能在当前轮结束前后受控硬停并完整落盘。
- **SC-003**: 报告中的每条结论都可追溯到 `runs/<run_id>/` 下的证据文件。
- **SC-004**: 单次 run 的 token 与成本可观测、可审计。

## 工程完成定义（DoD）

1. 每条 FR 均有对应验证步骤（测试或手工验收脚本）。
2. 契约校验通过：输入输出满足 `contracts.md`，错误响应符合统一结构。
3. 失败可见：失败迭代与日志完整保留，不允许“只展示最佳结果”。
4. 可复现：可基于 `runs/<run_id>/` 重放关键结果。
5. 文档同步：行为变化必须同步更新本文件与 `contracts.md`。
