# FARS Project Constitution

## Core Principles

### I. 安全优先执行（Safety-First Execution）
系统允许 LLM 生成和修改实验代码，但默认视为不可信执行。所有实验必须在受限环境中运行（容器/沙箱/资源限额），并具备超时、重试和强制终止机制。

### II. 证据驱动结论（Evidence Before Claims）
任何研究结论都必须绑定证据来源（实验结果文件、日志、文献条目）。没有证据链的结论只能标记为“假设”，不得进入最终摘要。

### III. 可复现优先（Reproducibility by Default）
每次运行必须落盘完整上下文：输入、配置、模型版本、提示词、日志、产物与摘要。关键实验应可由同配置重跑并得到同等级结论。

### IV. 人类闸门（Human-in-the-Loop Gate）
系统是“科研助理”而非“自动投稿器”。下列动作必须人工确认：研究方向最终选定、关键结论采纳、对外发布/投稿。

### V. 成本与时限约束（Budget & Time Bound）
每次任务必须设置预算上限、token 上限和执行时限。超过阈值自动降级或中止，防止无限探索。

## Additional Constraints

### 安全与合规约束
1. 禁止执行高风险系统命令与未授权网络访问。
2. 禁止提交真实密钥、私有数据、受限数据到公共仓库。
3. 默认禁用自动安装未知依赖；新增依赖需记录理由与来源。

### 科研质量约束
1. 结果必须区分“观察事实”和“解释推断”。
2. 文献引用需可检索、可回链，不允许伪造引用。
3. 对负结果和失败实验必须保留记录，不能只保留“成功样本”。

### 交付物约束
每次主流程运行最少输出：
1. `experiment_result.json`
2. `evidence_table.md`
3. `paper_draft.md`
4. `review_report.md`
5. `run_summary.json`

## Development Workflow & Quality Gates

### 文档先行
开发前必须先完成：
1. 本宪法（治理边界）
2. 用户故事文档（业务价值与验收）
3. Feature spec（需求、约束、成功标准）
4. task 拆分（可执行工作项）

### Stage Gate（必须逐关通过）
1. Gate A: Spec 完整性  
要求：用户故事、FR、成功指标、边界条件齐全。
2. Gate B: 执行安全性  
要求：超时、预算、日志、失败处理就位。
3. Gate C: 结果可追溯性  
要求：结论到证据一一对应。
4. Gate D: 人工复核  
要求：Reviewer 报告可读，人工确认后才能进入下一迭代或对外输出。

### 变更管理
1. 任何影响“安全、证据链、预算控制”的改动必须更新相关文档。
2. 宪法和故事文档更新后，必须同步检查 spec 与 tasks 是否失配。

## Governance

1. 本宪法优先级高于临时开发习惯与口头约定。
2. 每个 PR/提交都应显式说明是否满足本宪法 5 条核心原则。
3. 对违反核心原则的实现，默认拒绝合并，除非有书面豁免与回补计划。
4. 宪法修订需记录：修订动机、影响范围、迁移计划、回滚方案。
